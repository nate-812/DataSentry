# DataSentry 工程语言规范设计

## 1. 状态

- 日期：2026-06-25
- 状态：已确认，待实施
- 适用范围：整个 DataSentry 仓库，包括代码、数据库、API、配置、文档、Git 和用户界面

## 2. 目标

DataSentry 面向中文开发者和业务人员，代码结构必须符合通用英文工程规范，同时让业务说明、用户提示和维护注释保持中文可读。

本规范解决两个问题：

1. 避免代码标识符出现中文、拼音或拼音与英文混合命名。
2. 避免面向中文使用者的注释、帮助和异常消息全部使用英文。

## 3. 必须使用英文的内容

以下内容属于技术标识符或机器契约，必须使用语义清晰、拼写正确的英文：

- 包名、模块名和文件名。
- 类名、方法名、函数名。
- 变量名、参数名、常量名。
- 数据库名、表名、字段名和索引名。
- JSON、DTO、VO 和 API 字段名。
- 接口 URL 和路由名称。
- 配置项名称和环境变量名称。
- 枚举名称及其机器值。
- Git 分支名。
- 日志结构化字段名和稳定错误码。
- 测试函数、Fixture 和测试文件标识符。

禁止：

- 中文技术标识符。
- 拼音命名，例如 `yonghu_id`。
- 拼音与英文混合命名，例如 `user_xinxi`。
- 为了中文可读性修改已经对外或持久化的机器契约。

## 4. 应使用中文的内容

以下内容面向开发者、用户或业务人员阅读，应使用简洁、自然的中文：

- 代码注释和 docstring。
- CLI 命令说明、参数帮助和错误提示。
- 异常 `message`。
- 用户可见的日志事件文本。
- 模拟巡检中的结论、证据摘要、影响、建议和未知项。
- README、接口文档、设计文档和业务说明。
- 页面按钮、菜单、表单标题和提示信息。
- Git 提交说明中 Conventional Commits 类型之后的描述。

示例：

```text
feat: 增加模拟巡检 CLI
fix: 关闭 SQLite 迁移连接
docs: 明确工程语言规范
```

稳定错误码、JSON 字段和日志字段仍保持英文，例如：

```json
{
  "code": "storage.inspection_not_found",
  "message": "未找到指定巡检记录",
  "details": {
    "inspection_id": "..."
  }
}
```

## 5. 技术名词

行业通用技术名词保留英文原名，不进行生硬翻译，包括但不限于：

- Spring Boot
- Vue
- MySQL
- Kafka
- Flink
- Redis
- Doris
- SQLite
- API
- DTO
- VO
- Controller
- Service
- Mapper
- Repository
- CLI
- JSON
- UUID
- UTC

中文说明中可直接使用这些英文名词，例如“通过 Repository 读取 SQLite 中的巡检记录”。

## 6. M0 现有代码调整范围

本次只调整可读文本，不修改技术契约：

- 将 `src/datasentry/` 中英文模块 docstring、类 docstring、函数 docstring 改为中文。
- 将领域模型校验错误 message 改为中文。
- 将存储层错误 message 改为中文，稳定错误码保持不变。
- 将 CLI 帮助、命令说明、内部错误提示和模拟巡检业务文本改为中文。
- 将用户可见日志事件文本改为中文；结构化字段名保持英文。
- 更新测试断言，验证关键 CLI 和异常消息为中文。
- 在 `AGENTS.md` 中固化本规范。
- 更新 `docs/PROJECT_STATUS.md`，记录该工程决策。

以下内容不改：

- Python 标识符。
- 文件和目录名称。
- SQLite 表、字段和索引名称。
- CLI 命令名和参数名。
- JSON 字段。
- 枚举值。
- 稳定错误码。
- Git 历史中的既有英文提交说明；不重写已发布或本地历史。

## 7. 测试与验收

实施后必须满足：

1. 关键 CLI `--help` 输出包含中文说明。
2. `inspection show` 查不到记录时，`message` 为中文，错误码不变。
3. 模拟巡检的结论、证据摘要、影响、建议和未知项为中文。
4. 领域模型时间校验等异常 message 为中文。
5. 所有 Python 标识符、数据库契约和配置项保持英文。
6. Ruff、mypy 和全量 pytest 通过，覆盖率不低于 90%。
7. `git diff --check` 通过。

## 8. 后续约束

新增代码评审时同时检查：

- 标识符是否为规范英文。
- 是否出现拼音或混合命名。
- 用户可见文本是否为中文。
- 技术名词是否保留英文原名。
- 是否误改稳定错误码、API 字段或数据库契约。
