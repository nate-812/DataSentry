# M7.2 运维可用化设计

## 背景

M7.1 已补齐有限自治本地控制面，真实 K 线只读巡检也已通过一次性 Doris secret 注入闭合。当前暴露出的主要问题不是工具缺能力，而是运维入口还不够顺手：本地 DataSentry 进程需要 `DATASENTRY_*` 变量，云端 StreamLake 作业使用 `DORIS_PASSWORD`、`MYSQL_PASSWORD` 和 `REDIS_PASSWORD`，两套命名之间缺少安全、明确、可复用的说明与预检。

## 目标

- 提供一个只读、无网络副作用的 CLI 预检入口，解释目标配置和 secret 注入状态。
- 让缺 secret 的巡检输出更容易判断：这是本地进程环境缺变量，不是云端链路必然故障。
- 增加运维手册，固化本地真实只读 smoke 的安全运行方式。
- 仓库只保存变量名、映射关系和命令模板，不保存真实密码、Token、私钥或连接串。

## 非目标

- 不实现真实生产写操作。
- 不自动读取 `/root/.streamlake-secrets`。
- 不把 `streamlake_writer` 变成 DataSentry 的长期默认账号。
- 不新增任意 Shell、自动重启、自动补数、自动 Savepoint 恢复或自动改配置能力。

## 方案

新增 `datasentry ops preflight` 命令，读取 `config/targets.toml` 或 `DATASENTRY_TARGETS_FILE` 指向的目标配置。命令输出 JSON，包含：

- 目标配置路径与加载状态。
- Doris/MySQL/Redis/SSH 目标声明的 secret 环境变量名。
- 每个变量当前是否在进程环境中设置，只显示 `configured` 或 `missing`，不显示值。
- 对常见云端变量名的提示，例如 `DORIS_PASSWORD` 可映射到 `DATASENTRY_DORIS_PASSWORD`。

增强 CLI 错误输出：当底层错误携带 `configuration.secret_missing` 时，在安全错误 JSON 中保留缺失变量名和中文说明。真实 secret 值仍然不进入 stdout、stderr、SQLite 或日志。

新增 `docs/operations/live-smoke.md`，说明下次运维时如何选择：

1. 长期方案：为 DataSentry 准备专用只读账号和 `DATASENTRY_*` 变量。
2. 应急方案：用一次性进程环境把云端 root-only secret 映射成 `DATASENTRY_*`，运行后不落盘。
3. 判断范围：K 线巡检通常需要 Doris，不一定触发 MySQL/Redis；风控和黑名单问题才会触发 MySQL/Redis。

## 验证

- 单元测试覆盖 secret 状态只显示变量名和状态，不泄露值。
- CLI 场景测试覆盖 `ops preflight` 的 JSON 输出。
- CLI 场景测试覆盖缺 Doris secret 时输出包含变量名和可读建议，但不包含 secret 值。
- 运行 Ruff、mypy、pytest，并更新 README 与项目状态文档。
