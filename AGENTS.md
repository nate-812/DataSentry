# DataSentry Agent 工程协作规则

本文件适用于整个仓库。Codex 在本项目中开发、测试和维护代码时，应持续遵守以下规则。

## Git 与 GitHub

- `origin` 指向 `https://github.com/nate-812/DataSentry.git`。
- `main` 保持可运行、可回退；较大功能和高风险修改使用独立分支。
- 在完成一个连贯、可验证的工作单元后创建提交，提交信息使用简洁的 Conventional Commits 风格，例如 `feat: ...`、`fix: ...`、`docs: ...`、`chore: ...`。
- 提交前检查 `git diff`、运行与改动相称的测试或验证，并明确记录未验证项。
- 在稳定检查点按需推送到 GitHub，避免大量未同步的本地提交。长任务至少在每个可恢复里程碑推送一次。
- 大功能优先通过 Pull Request 合并；小型文档、配置和仓库维护可直接提交到 `main`。
- 未经用户明确批准，不得强制推送、重写已发布历史、删除远端分支、合并 Pull Request 或执行破坏性 Git 命令。
- 推送前先拉取并检查远端变化；存在分叉或冲突时停止并说明，不擅自覆盖。
- 不把失败、明显不完整或未经说明的状态当作稳定备份推送。需要保存探索性工作时，使用清楚命名的功能分支或 WIP 提交。

## 安全与隐私

- 不提交密码、Token、Cookie、私钥、AK/SK、真实 `.env`、云凭据或生产连接字符串。
- 提交前检查新增文件和 diff 是否包含秘密；示例配置只能使用占位值。
- 不读取或提交 Shell History、SSH 私钥、系统钥匙串及工作区外的敏感文件。
- 运行日志、诊断输出和提交内容中的秘密必须脱敏。

## 工程质量

- 修改前先理解现有结构和相关知识文档，保持改动聚焦。
- 新功能和缺陷修复应配套适当测试；提交前运行最小相关测试集，并在风险较高时扩大验证范围。
- 不为了顺手而进行无关重构，不覆盖用户已有但未提交的改动。
- 完成工作时报告：改动摘要、验证结果、提交哈希、当前分支及 GitHub 同步状态。

## StreamLake 运维 Agent 边界

- 第一版聚焦主题知识加载、白名单只读查询、SQLite 巡检快照、证据化诊断和缺失组件提醒。
- 第一版不实现 RAG、任意 Shell、自动重启、自动补数、自动改配置或自动 Savepoint 恢复。
- 实时状态必须通过受限只读工具现场查询；历史文档不得包装成当前事实。
- 任何写操作都必须遵循风险分级、审批、回滚或恢复方案以及操作后验证。
